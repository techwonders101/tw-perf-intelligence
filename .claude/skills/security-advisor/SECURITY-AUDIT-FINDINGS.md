## Findings

### Medium Severity

#### M1: OAuth Callback Lacks Nonce Verification
**Location**: [class-st-admin.php:87-90](admin/class-st-admin.php#L87-L90)
**Type**: Missing CSRF Protection
**Status**: **Resolved** (Feb 2026)

**Description**:
The OAuth callback handler at `handle_stokkap_callback()` doesn't verify a nonce. While this is an OAuth callback from an external service, it accepts credentials via GET parameters.

**Current Code**:
```php
if (isset($_GET['page']) && $_GET['page'] === 'stokkap-settings'
    && isset($_GET['action']) && $_GET['action'] === 'connected') {
    $this->handle_stokkap_callback();
}
```

**Original Risk**:
An attacker could potentially craft a malicious link to inject their own credentials if they can convince an admin to click it.

**Resolution**:
The token exchange pattern implemented in the OAuth flow effectively mitigates this concern:
1. The `connection_token` is generated by Stokkap's server, not the WordPress site — an attacker cannot generate arbitrary tokens
2. The token is one-time use and short-lived — it's exchanged server-side for API credentials via a POST to Stokkap's API
3. Even if an attacker crafted a link with their own token, the result (linking their Stokkap account) would be immediately visible in the admin UI and reversible by disconnecting
4. The attack chain requires the victim admin to click a link AND the attacker to have a valid, unused connection token — making practical exploitation unrealistic

No code changes needed — the existing token exchange architecture provides sufficient protection.

### Low Severity

#### L1: Sensitive Credentials in URL During OAuth
**Location**: [class-st-admin.php:338-341](admin/class-st-admin.php#L338-L341)
**Type**: Information Exposure
**Status**: **Resolved** (token exchange pattern)

**Description**:
OAuth credentials were previously passed via GET parameters which could be logged in server access logs and browser history.

**Resolution**:
The OAuth flow now uses a token exchange pattern. Instead of passing `client_id` and `client_secret` directly in the URL, Stokkap redirects back with a short-lived `connection_token`. The WordPress plugin then exchanges this token server-side (via POST) for the actual API credentials. The connection token is one-time use and short-lived, so even if it appears in server logs, it cannot be reused.

---

#### L2: Debug Logging May Expose Sensitive Data
**Location**: Multiple files using `stokkap_log()`
**Type**: Information Exposure
**Status**: **Resolved** (Feb 2026 fix #5)

**Description**:
When debug logging was enabled, various sensitive operations were logged including API credentials creation:

```php
stokkap_log('Consumer key starts with: ' . substr($api_credentials['consumer_key'] ?? 'NULL', 0, 10));
```

**Resolution**:
Feb 2026 fix #5 removed all sensitive credential logging:
- Removed token prefix logging (`substr($connection_token, 0, 10)`)
- Removed full response body logging from connection attempts
- Removed consumer key and payload key logging during credential exchange
- Removed full webhook payload logging in debug mode

### Fix Details

#### #2 — Escape API Response Data in Updater

**File:** `includes/licensing/class-techwonders-updater.php`

External API response fields (`name`, `version`, `tested`, `requires`, `requires_php`) sanitized with `sanitize_text_field()` before being stored in the plugin update transient. HTML sections (`description`, `changelog`) sanitized with `wp_kses_post()` to allow safe HTML subset. Prevents stored XSS if the licensing API were compromised.

#### #3 — Add Capability Checks to AJAX Handlers

**File:** `admin/class-st-admin.php`

Two AJAX handlers (`ajax_recalc_status`, `ajax_check_connection`) had nonce verification but no capability check. Added `current_user_can('manage_woocommerce')` with early `wp_send_json_error()` return. Defence in depth — nonces alone don't guarantee the user has appropriate permissions.

#### #4 — Sanitize External Error Messages

**File:** `admin/class-st-admin.php`

Replaced verbatim external error messages (from Stokkap API connection attempts) with a generic user-facing message. The raw error message is still logged server-side (with `sanitize_text_field()`) for debugging. Prevents information disclosure of internal API endpoints, server errors, or stack traces.

#### #5 — Stop Logging Sensitive Credentials

**Files:** `admin/class-st-admin.php`, `includes/class-st-webhooks.php`

Removed:
- Token prefix logging (`substr($connection_token, 0, 10)`)
- Full response body logging from connection attempts
- Consumer key and payload key logging during credential exchange
- Full webhook payload logging in debug mode

These could leak sensitive data to log files accessible to hosting providers or other plugins.

#### #6 — Add Advisory Lock to Stock Depletion

**File:** `includes/class-st-stock.php`

Wrapped `deplete_stock()` in a MySQL advisory lock using the pattern already established in `prevent_concurrent_stock_reduction()`:

```php
$lock_name = $wpdb->prefix . 'deplete_stock_' . $product_id;
$locked = $wpdb->get_var($wpdb->prepare("SELECT GET_LOCK(%s, 5)", $lock_name));
// ... stock depletion logic in try/finally ...
$wpdb->get_var($wpdb->prepare("SELECT RELEASE_LOCK(%s)", $lock_name));
```

Per-product lock with 5-second timeout. Uses `try/finally` to guarantee lock release. Prevents race conditions when multiple concurrent orders reduce stock for the same product.

#### #8 — Validate Webhook URL Against Private IPs

**File:** `includes/class-st-webhooks.php`

Added `is_safe_webhook_url()` method called before every webhook send:
- Requires HTTPS scheme
- Blocks `localhost`, `127.0.0.1`, `::1`, `0.0.0.0`
- Resolves hostname via `gethostbyname()` and blocks private/reserved IP ranges
- Uses `FILTER_VALIDATE_IP` with `FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE`

Prevents SSRF attacks where an attacker could set `app_url` to an internal service.

#### #9 — Fix JSON Input Handling Order

**File:** `admin/class-st-quick-lookup.php`

Changed from:
```php
$raw_items = sanitize_text_field(wp_unslash($_POST['items']));
```

To:
```php
$raw_items = wp_unslash($_POST['items']);
```

`sanitize_text_field()` converts `&` to `&amp;` and strips tags, which corrupts JSON structure. The JSON is decoded immediately after, and individual values are sanitized in the foreach loop using `absint()`, `sanitize_key()`, and `intval()`.

#### #10 — Replace Inline onclick Handler

**File:** `admin/js/admin.js`

Replaced inline `onclick="return confirm(...)"` which injected `stLocations.i18n.confirmDelete` directly into an HTML attribute. If the i18n string contained a single quote, it could break out of the `confirm()` call.

Changed to a delegated jQuery event handler:
```javascript
$(document).on('click', '.st-confirm-delete-location', function(e) {
    if (!confirm(stLocations.i18n.confirmDelete || 'Are you sure you want to delete this?')) {
        e.preventDefault();
    }
});
```

The i18n string is now only used as a JavaScript variable, never injected into HTML.
